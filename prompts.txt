# Project-Specific Notes (Dvenue)

- This is a **Flutter** frontend (not pure web/Next.js), therefore:
  - Primary client integration uses the official `razorpay_flutter` package (supports Android + iOS + limited web).
  - For Flutter web builds we still support the checkout.js approach via JS interop if needed.
- We use a **dedicated payments/** microservice (port 4001) separate from the main gateway backend (port 4000).
- Payments server:
  - Fetches authoritative amounts from **Firebase Firestore** (bookings + listings collections) — never trusts client amount.
  - Stores every payment + premium/subscription state in **Supabase**.
  - Updates booking/listing paid/promoted flags back in Firebase after successful verify.
- "Advertisement" payments (premium listing boosts) are supported via the same `type` field.
- Main backend acts purely as a proxy/gateway. Flutter never talks directly to payments or Supabase/Firebase.
- See payments/arch.md for the complete data flow diagram and security model.
- See payments/razorpay.handler.js for the actual create + HMAC verify implementation.
